# 自頂向下分析：行程排程與並行性

SystemC 中的並行性是 **模擬的** (除非使用現代化的平行擴展)。模擬核心透過在行程之間快速切換來創造平行硬體執行的假象。

## 1. 行程型別
SystemC 提供兩種基本的行程型別，各自服務於特定的建模需求。

### 1.1 方法行程 (`SC_METHOD`)
-   **抽象 (Abstraction)**: 組合邏輯 (Combinational Logic) / 反應式軟體 (Reactive Software)。
-   **機制**: 標準的 C++ 成員函式。
-   **堆疊 (Stack)**: 在核心的堆疊上執行 (或在觸發它的執行緒的堆疊上)。
-   **行為**:
    -   必須執行到完成。
    -   **不能** 呼叫 `wait()`。
    -   在呼叫之間不保留區域狀態 (除非儲存在類別成員中)。
-   **使用案例**: 實作類 RTL 的布林邏輯、簡單算術運算或處理中斷。

### 1.2 執行緒行程 (`SC_THREAD`)
-   **抽象**: 序列邏輯 (Sequential Logic) / 測試平台 (Testbenches) / 軟體任務。
-   **機制**: 一個 **協程 (Coroutine)** (使用者層級執行緒)。
-   **堆疊**: 分配一個私有堆疊 (例如 64KB)。
-   **行為**:
    -   可以使用 `wait()` 暫停執行。
    -   在暫停之間保留區域變數狀態。
    -   本質上 "永久" 執行 (通常是無限的 `while(true)` 迴圈)。
-   **使用案例**: 撰寫測試序列 (例如 "驅動 A，等待 10ns，驅動 B")、複雜的狀態機，或高階演算法模型。

### 1.3 時脈執行緒 (`SC_CTHREAD`)
-   **抽象**: 可合成的同步邏輯 (Synthesizable Synchronous Logic)。
-   **機制**: 一個特化的 `SC_THREAD`，*僅* 對時脈邊緣敏感。
-   **遺留 (Legacy)**: 在現代用法中已大部分被 `SC_THREAD` 或 `SC_METHOD` 取代，但在歷史上對高階合成 (HLS) 很重要。

## 2. 排程器的視角
排程器 (`sc_simcontext`) 不 "知道" 使用者程式碼做了什麼。它只看到 `sc_process_b` 的處理器。

### 2.1 可執行佇列 (Runnable Queue)
-   排程器維護一個在當前 delta cycle 中準備執行的行程清單 (或佇列)。
-   **非搶占式 (Non-Preemptive)**: 一旦行程開始 (方法或執行緒)，排程器 **無法** 中斷它。行程必須自願讓出 (方法回傳，執行緒呼叫 `wait()`)。
-   **順序**: 同一 delta cycle 內行程的執行順序在標準中是 **未定義的**。這強化了使用訊號 (更新階段) 進行通訊的紀律，以避免競態條件。

### 2.2 上下文切換 (執行緒)
當 `sc_simcontext` 挑選一個 `SC_THREAD` 來執行時：
1.  **儲存核心**: 它將當前的 CPU 暫存器 (指令指標、堆疊指標) 儲存到核心的上下文中。
2.  **恢復執行緒**: 它從執行緒的上下文區塊中載入已儲存的暫存器。
3.  **恢復**: CPU 跳轉到使用者執行緒中上一次 `wait()` 呼叫之後的指令。
4.  **暫停**: 當使用者呼叫 `wait()` 時，執行相反的操作。

此上下文切換是透過 "協程套件 (Coroutine Package)" (`sc_cor_pkg`) 實作的，它封裝了底層機制，如：
-   `setjmp`/`longjmp` (QuickThreads)
-   Windows Fibers
-   Pthreads (用於標準 C++ 實作)

## 3. 關鍵檔案
-   `sysc/kernel/sc_process.cpp`: 管理行程狀態 (暫停、正常、殭屍) 的基底類別。
-   `sysc/kernel/sc_method_process.cpp`: 輕量級方法執行器的實作。
-   `sysc/kernel/sc_thread_process.cpp`: 基於堆疊的執行緒的實作。
-   `sysc/kernel/sc_cor.h`: 低階上下文切換器的介面。
