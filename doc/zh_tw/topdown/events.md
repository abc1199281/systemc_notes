# 自頂向下分析：事件系統

SystemC 是一個 **事件驅動模擬器 (Event-Driven Simulator)**。事件是行程之間同步和通訊的基本機制。

## 1. `sc_event` 的本質
`sc_event` 是一個 **無狀態、瞬時** 的通知。
-   **沒有值**: 它不攜帶資料。
-   **沒有持續時間**: 它只在通知的瞬間存在。
-   **沒有記憶**: 如果行程在事件發生 *之後* 才等待它，它就會錯過 (除非使用 `event_queue` 或旗標模式)。

## 2. 通知型別
事件可以透過三種方式觸發，對應於模擬迴圈的不同階段：

### 2.1 立即通知 (Immediate Notification)
-   **語法**: `e.notify()`
-   **效果**: 事件在當前評估階段 **立即** 觸發。
-   **使用案例**: 在同一 delta cycle 中執行的行程之間進行類軟體的同步。
-   **危險**: 如果與共享變數一起使用，可能會引入非確定性 (誰先執行/最後執行的競態條件)。

### 2.2 Delta 通知
-   **語法**: `e.notify(SC_ZERO_TIME)`
-   **效果**: 事件被排定在 **下一個 delta cycle 的開始** (更新/通知階段) 時觸發。
-   **使用案例**: 這是 **基本通道 (Primitive Channels)** (如 `sc_signal`) 的標準機制。當訊號獲得新值時，它請求 delta 通知以在 *所有更新完成後* 喚醒敏感行程。這確保了確定性。

### 2.3 計時通知 (Timed Notification)
-   **語法**: `e.notify(10, SC_NS)`
-   **效果**: 事件被排定在模擬時間推進指定量時觸發。
-   **使用案例**: 建模傳輸延遲、時脈週期或超時。

## 3. 動態敏感度 (Dynamic Sensitivity)
行程可以使用 `wait()` (執行緒) 或 `next_trigger()` (方法) 動態地等待事件。

-   **`wait(e)`**: 暫停執行緒直到 `e` 被通知。
-   **`wait(e1 | e2)`**: 暫停直到 `e1` 或 `e2` **其中之一** 被通知 (OR 清單)。
-   **`wait(e1 & e2)`**: 暫停直到 **兩者都** 被通知 (AND 清單)。這需要核心追蹤此特定 `wait` 呼叫的通知狀態。

## 4. 核心互動
當呼叫 `notify()` 時：
1.  **立即**: 核心立即遍歷事件的敏感行程清單 (`m_methods_dynamic`, `m_threads_dynamic`) 並將它們推送到 **可執行佇列**。
2.  **Delta/計時**: 事件被加入核心的 **事件佇列**。當排程器到達適當的階段時，它 "觸發" 事件，重複步驟 1。

## 5. 硬體映射
| SystemC 概念 | Verilog 對應 |
| :--- | :--- |
| `sc_event` | `event` (命名事件) |
| `sc_event.notify()` | `-> e;` |
| `sc_event.notify(0)` | 在下一個 NBA 區域觸發的邏輯 (概念上) |
| `wait(e)` | `@(e)` |
